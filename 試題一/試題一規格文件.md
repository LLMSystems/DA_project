# 試題一規格文件

本文件整理 `interview_assignment.pdf` 對試題一的要求，以及目前已完成的網站操作與頁面行為調查結果，作為後續實作爬蟲、資料清理、CSV 輸出、資料庫寫入與錯誤處理的依據。

## 1. 題目目標

開發一支爬蟲程式，針對內政部戶政司網站的村里街路門牌異動查詢功能，依指定條件擷取資料，先輸出檔案，再寫入資料庫，並保留成功與失敗 Log。

## 2. 題目指定查詢條件

- 縣市：`台北市`
- 區域：`各區`
- 編訂日期：`民國 114/09/01 ~ 114/11/30`
- 編訂類別：`門牌初編`

## 3. 查詢入口與操作流程

### 3.1 入口網址

- 入口頁：`https://www.ris.gov.tw/app/portal/3053`

### 3.2 正確功能路徑

入口頁有兩組主要功能：

1. `現有村里街路門牌查詢`
2. `村里街路門牌異動查詢（資料起始日期：87年8月1日）`

依題目描述，正確路徑為：

1. 進入入口頁
2. 找到 `村里街路門牌異動查詢`
3. 點選 `以鄉鎮市區、編釘類別查詢`

### 3.3 頁面切換行為

- 點選 `以鄉鎮市區、編釘類別查詢` 後，網址仍維持：
  `https://www.ris.gov.tw/app/portal/3053`
- 頁面切換成 `請點選欲查詢縣市`
- 縣市選擇方式是互動地圖，不是一般表單欄位

### 3.4 選縣市後的查詢頁

- 點選 `台北市` 後，網址變為：
  `https://www.ris.gov.tw/info-doorplate/app/doorplate/query`
- 頁面進入正式查詢表單

## 4. 查詢表單規格

### 4.1 欄位列表

查詢表單目前已確認欄位如下：

1. `縣市別`
   - 顯示值：`臺北市`
   - 非手動輸入欄位，為前一步縣市選擇結果

2. `鄉(鎮市區)`
   - 類型：下拉選單
   - 必填

3. `編訂日期起訖時間`
   - 類型：日期選擇器
   - 必填
   - 分成 `從` 與 `到` 兩個欄位

4. `含未記載編訂日期資料`
   - 類型：checkbox
   - 非必填

5. `編訂類別`
   - 類型：下拉選單
   - 題目指定值為 `門牌初編`

6. `圖形驗證碼`
   - 類型：captcha
   - 必填

### 4.2 按鈕與互動元件

已確認按鈕包含：

- `搜尋`
- `重填`
- `產製新驗證碼`
- `語音播放`

頁面並標示：

- `標註星號（*）表示此欄位必須填寫`

## 10. 建議技術方案

根據目前頁面行為，建議 `試題一` 採用 `Selenium` 為主，理由如下：

1. 入口流程包含互動地圖點擊
2. 查詢頁包含 datepicker
3. 驗證碼需處理人工輸入
4. 查詢結果為同頁局部更新
5. 查詢結果包含分頁與每頁筆數切換
6. `table` 結構明確，利於逐列解析

相較之下，單純 `requests + BeautifulSoup` 雖可能抓到部分 HTML，但不利於穩定操作以下流程：

- 地圖選縣市
- datepicker 選日期
- captcha 互動
- 分頁局部更新等待

### 10.1 技術選型說明

本作業的爬蟲主體採用 `Selenium`，資料儲存採用 `aiosqlite`，API 服務採用 `FastAPI`。

之所以選擇 `Selenium`，是因為目標網站並非單純靜態頁面，而是包含以下互動行為：

1. 需先經由互動式地圖選取縣市
2. 日期欄位需透過 datepicker 操作，無法直接手打
3. 驗證碼必填，且需配合人工輸入
4. 查詢結果在同頁局部更新，不是單純頁面跳轉
5. 結果包含分頁與每頁筆數切換

相較於 `requests` 或 `Scrapy`，`Selenium` 更適合穩定模擬實際使用者操作流程，也較容易處理前端互動元件與結果頁狀態變化。

資料庫部分選用 `SQLite` 作為底層儲存，並以 `aiosqlite` 操作，主要考量如下：

1. `SQLite` 部署簡單，不需額外安裝資料庫服務
2. 審閱者 clone 專案後可直接執行驗證，重現成本低
3. `aiosqlite` 可讓後續 API 與查詢紀錄功能維持非同步風格
4. 與 `FastAPI` 的 async 路由設計較一致

雖然爬蟲主流程因 `Selenium` 特性仍以同步操作為主，但在資料寫入與 API 查詢層採用 `aiosqlite`，可兼顧實作簡單性與整體技術風格一致性。

API 服務選擇 `FastAPI`，原因如下：

1. 撰寫速度快，適合本作業時限
2. 輸入輸出模型清楚，便於定義查詢介面
3. 易於測試與展示 API 執行結果
4. 後續若需擴充查詢紀錄、異常通知或健康檢查端點，延展性佳

整體而言，`Selenium + aiosqlite + FastAPI` 能兼顧可實作性、可維護性與作業驗證便利性。

## 12. 建議執行流程

後續實作爬蟲時，可依下列順序進行：

1. 開啟入口頁
2. 點選 `村里街路門牌異動查詢`
3. 點選 `以鄉鎮市區、編釘類別查詢`
4. 在互動地圖選 `台北市`
5. 進入查詢頁後，逐區執行：
   - 選擇區別
   - 使用 datepicker 選起始日期
   - 使用 datepicker 選結束日期
   - 選擇 `門牌初編`
   - 要求使用者輸入驗證碼
   - 送出查詢
6. 若出現 `查無資料`
   - 記錄查詢條件與結果狀態
   - 繼續下一區
7. 若驗證碼錯誤
   - 記錄錯誤
   - 重新取得驗證碼或要求重新輸入
8. 若有資料
   - 將每頁筆數改為 `200`
   - 逐頁抓取表格
   - 合併所有頁面資料
9. 各區完成後：
   - 清理資料
   - 輸出 CSV
   - 寫入資料庫
   - 紀錄成功 Log

## 13. 錯誤處理重點

至少應處理以下情境：

1. 驗證碼錯誤
2. 查無資料
3. 表格載入失敗
4. 分頁點擊後資料未更新
5. 頁面元素找不到
6. 網站結構調整
7. 網路請求失敗
8. 資料庫寫入失敗
9. CSV 寫檔失敗

## 14. 目前已知的重要實作結論

1. 本題不適合當作純靜態頁面爬取處理。
2. `台北市各區` 必須以逐區方式完成。
3. 日期要透過 datepicker 選取。
4. 驗證碼可採人工輸入流程。
5. 查詢結果需支援多頁擷取。
6. 查詢結果應優先把每頁筆數調到 `200`。
7. `變更前地址` 欄位可能為空值，資料模型需允許空白。
